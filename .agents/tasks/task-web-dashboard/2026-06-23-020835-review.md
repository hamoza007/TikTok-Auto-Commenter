# Flask web dashboard for TikTok Auto Commenter

This change converts the existing CLI-based TikTok auto-commenter into a Flask web application. It adds a SQLite-backed dashboard with three sections: Settings (CRUD for TikTok accounts with session IDs, proxies, and OpenAI keys), Dashboard (project creation linking videos to comment templates and accounts, with background execution), and History (per-comment result log). The app reuses the original `TikTok` class from `main.py` for the actual API calls and adds optional OpenAI-powered comment generation.

**Watch for:** Credentials stored in plaintext in the database and rendered in plaintext in the browser (confirmed). The proxy field stored on each account is never actually passed to the `TikTok` class or the underlying `requests` calls (confirmed). Background threads share the SQLAlchemy session unsafely and have no timeout or cancellation mechanism (likely). The hardcoded `SECRET_KEY` means session cookies are trivially forgeable in production (confirmed).

## High-level view

The app factory in `app.py` defines three SQLAlchemy models and groups of routes behind a single Flask instance. Account, Project, and History form a simple FK chain with cascade deletes. The account model stores credentials and configuration; the project model stores a video target plus comment text; history records one row per comment attempt. Projects are executed in daemon threads that import the original `TikTok` class, iterate over newline-separated comments, and record success/failure per comment.

The proxy field is accepted in the UI and persisted in the database but never injected into the HTTP client. The `TikTok` class doesn't accept a proxy parameter, and `run_project_task` doesn't attempt to pass one. Any user who configures a proxy expecting it to be used will get direct connections instead.

OpenAI integration is a thin wrapper around `gpt-3.5-turbo` completions. It fires once per comment line, using the template text as a prompt seed. Failures fall back silently to the original template text. The API key is stored and rendered in cleartext.

The templates use Bootstrap 5 via CDN with a dark theme. Forms use standard POST with page-reload; no JavaScript beyond Bootstrap's modal handling. The settings page renders full API keys and session IDs in edit-modal input fields, which means they appear in the DOM of any browser with access to the page.

1. **Proxy never used** — `Account.proxy` is stored but never passed to `TikTok` or `requests`. Either plumb it through or remove the field from the UI to avoid false confidence.
2. **Credentials in cleartext** — Session IDs and OpenAI API keys are stored in SQLite without encryption and rendered in full in the HTML. Mask display values and consider application-level encryption at rest.
3. **Hardcoded SECRET_KEY** — The fallback `"dev-secret-key-change-in-production"` means anyone who doesn't set the env var gets forgeable session cookies. Fail loudly or generate a random key at startup.
4. **Thread-safety of SQLAlchemy sessions** — Background threads call `db.session` without scoped sessions. Concurrent project runs may corrupt the session or produce stale reads.
5. **No thread timeout or cancellation** — A spam-loop or network hang in `run_project_task` blocks the daemon thread indefinitely. No mechanism exists to stop or time-out a running project.
6. **Silent exception swallowing** — The outer `except Exception` in `run_project_task` sets status to "failed" but discards the error. No logging, no user-visible error detail.
7. **No CSRF protection** — All state-mutating routes accept bare POST with no token verification. Any page the user visits can submit forms to these endpoints.

## Proxy field is stored but never connected

The `Account` model stores `proxy` as a string, the UI collects it, and it persists in the database. But the execution path in `run_project_task` instantiates `TikTok(account.session_id)` — the `TikTok` class only accepts a session ID and builds its own `requests` calls with no proxy kwarg. There is no code that reads `account.proxy` and passes it to `requests.get()` or `requests.post()`.

This isn't just a missing feature; it's a correctness issue for users who assume their traffic is proxied. If the proxy is meant for future use, the UI should indicate it's not active. If it's meant to be functional now, the `TikTok` class needs a `proxies` parameter threaded through to all its HTTP calls.

## Credential exposure in templates and database

Session IDs and OpenAI API keys are stored as plaintext `db.String(255)` columns. The settings template renders them as `value="{{ account.session_id }}"` and `value="{{ account.openai_api_key }}"` inside edit modals. This means:

- Any XSS vector on the page exposes all credentials.
- The SQLite file (gitignored but on-disk) contains raw keys.
- Browser developer tools show credentials in the DOM even without an attack.

The settings list table masks the API key display (`{{ account.openai_api_key[:8] }}...`) but the edit modal reveals the full value. Session IDs are not masked anywhere.

## Hardcoded secret key and missing CSRF

```python
app.config["SECRET_KEY"] = os.environ.get("SECRET_KEY", "dev-secret-key-change-in-production")
```

The fallback string is committed to version control. Flask's session cookies are signed with this key, so anyone who runs the app without setting the env var has predictable cookie signatures. Combined with the absence of CSRF tokens on any form (Flask-WTF or manual), the app is vulnerable to cross-site form submission against all POST routes.

## Background execution: thread safety and lifecycle

`run_project_task` runs in a daemon thread created per project run:

```python
thread = threading.Thread(target=run_project_task, args=(app, project.id))
thread.daemon = True
thread.start()
```

The function uses `db.session` inside `with app.app_context()`. SQLAlchemy's default session is scoped per-thread only when using `scoped_session`; the plain `db.session` from Flask-SQLAlchemy 3.x uses the app-context-local session, which is not thread-safe when multiple background threads share the same app context simultaneously. If two projects run concurrently, they may interleave commits on the same underlying connection.

There is no mechanism to cancel a running project, no timeout on the `requests` calls inside `TikTok.send()`, and no limit on concurrent threads. A pathological case (many projects triggered rapidly, each hitting spam delays) would spawn unbounded threads all sleeping 10 seconds in a loop.

## Error handling gap in project execution

The entire execution body is wrapped in:

```python
except Exception as e:
    project.status = "failed"
    db.session.commit()
```

The exception `e` is captured but never logged, stored, or surfaced. The user sees "failed" status with no indication of why. For debugging and user feedback, the error message (or at least a category) should be persisted in the history or project record.

## aweme_id extraction fallback

```python
def extract_aweme_id(video_url):
    match = re.search(r'/video/(\d+)', video_url)
    if match:
        return match.group(1)
    if video_url.strip().isdigit():
        return video_url.strip()
    return video_url.strip()
```

The final fallback returns the entire URL string as the `aweme_id`. This will be passed to `TikTok.send()` as the `aweme_id` parameter, which casts it to `int(aweme_id)` in the request data. A non-numeric URL that doesn't match either pattern will cause a `ValueError` inside `TikTok.send()`, which propagates up and marks the project as failed. The extraction function should validate or reject URLs that can't be resolved to a numeric ID.
